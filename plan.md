## 1. Project Overview & Key Decisions

### What We Are Building

A production-grade clone of [Linear.app](https://linear.app) that satisfies every assignment requirement while delivering a significantly more impressive submission. Linear is the industry benchmark for developer task management — clean opinionated UI, fast interactions, well-thought-out data model. Building a focused clone demonstrates real-world product thinking, not just CRUD competence.

**Assignment requirements covered:**

| Requirement | Linear Clone Coverage |
|---|---|
| Auth (signup / login) | ✅ JWT with refresh token rotation |
| Project & team management | ✅ Workspaces + member roles |
| Task creation, assignment, status | ✅ Issues with priority, assignee, due date |
| Dashboard (tasks, status, overdue) | ✅ My Issues + overdue + summary counts |
| REST APIs + Database | ✅ Full Express API + PostgreSQL via Prisma |
| Proper validations & relationships | ✅ Zod schemas + FK constraints + cascades |
| Role-based access control | ✅ ADMIN / MEMBER at workspace level |
| Deploy on Railway | ✅ Single service, Postgres add-on |

### Scope — What We Clone vs. What We Skip

**In scope (core clone):**
- Workspaces — replaces "Project" in assignment terms
- Issues — title, description, status, priority, assignee, due date, auto-identifier (ENG-1, ENG-2…)
- Kanban board — columns: Backlog / Todo / In Progress / In Review / Done
- My Issues view — the dashboard requirement
- Team member management with Admin / Member roles
- Linear-style dark sidebar + clean list UI

**Explicitly out of scope** (won't cost points, would take weeks):
- Cycles / sprints
- GitHub integration
- Real-time sync (WebSockets)
- Notifications system
- Roadmap / timeline view
- Keyboard shortcut engine

### Module Architecture

Each backend module is fully self-contained. The controller handles HTTP (parsing, response shaping). The service holds all business logic and DB access. Controllers never call another module's controller — cross-module calls go through the service layer only.

| Module | Responsibility | Key Dependencies |
|---|---|---|
| auth | Register, login, refresh, logout, /me | bcryptjs, jwt.ts, prisma.ts |
| workspaces | CRUD workspaces, membership checks | authenticate, requireRole, prisma.ts |
| issues | CRUD issues, status/priority/assign updates | authenticate, requireRole, prisma.ts |
| members | Add/remove members, role changes | authenticate, requireRole (ADMIN only) |
| dashboard | Aggregated summary + overdue scoped to user | authenticate, prisma.ts |

## 3. Data Model

### Prisma Schema

```prisma
// prisma/schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id           String   @id @default(cuid())
  email        String   @unique
  name         String
  passwordHash String
  avatarUrl    String?
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  memberships    WorkspaceMember[]
  assignedIssues Issue[]          @relation("Assignee")
  createdIssues  Issue[]          @relation("Reporter")
  refreshTokens  RefreshToken[]
}

model Workspace {
  id        String   @id @default(cuid())
  slug      String   @unique  // e.g. "acme-corp" — used in all URLs
  name      String
  logoUrl   String?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  members WorkspaceMember[]
  issues  Issue[]
}

model WorkspaceMember {
  id          String        @id @default(cuid())
  workspaceId String
  userId      String
  role        WorkspaceRole @default(MEMBER)
  joinedAt    DateTime      @default(now())

  workspace Workspace @relation(fields: [workspaceId], references: [id], onDelete: Cascade)
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([workspaceId, userId])
}

model Issue {
  id          String      @id @default(cuid())
  identifier  String      // e.g. "ENG-42" — auto-generated, unique per workspace
  workspaceId String
  title       String
  description String?
  status      IssueStatus @default(TODO)
  priority    Priority    @default(MEDIUM)
  assigneeId  String?
  reporterId  String
  dueDate     DateTime?
  createdAt   DateTime    @default(now())
  updatedAt   DateTime    @updatedAt

  workspace Workspace @relation(fields: [workspaceId], references: [id], onDelete: Cascade)
  assignee  User?     @relation("Assignee", fields: [assigneeId], references: [id], onDelete: SetNull)
  reporter  User      @relation("Reporter", fields: [reporterId], references: [id])

  @@unique([workspaceId, identifier])
  @@index([workspaceId, status])
  @@index([assigneeId])
}

model RefreshToken {
  id        String   @id @default(cuid())
  token     String   @unique
  userId    String
  expiresAt DateTime
  createdAt DateTime @default(now())

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
}

enum WorkspaceRole { ADMIN  MEMBER }

enum IssueStatus {
  BACKLOG
  TODO
  IN_PROGRESS
  IN_REVIEW
  DONE
  CANCELLED
}

enum Priority {
  URGENT
  HIGH
  MEDIUM
  LOW
  NO_PRIORITY
}
```

### Relationships & Cascade Rules

| Relationship | Type | On Delete |
|---|---|---|
| User → WorkspaceMember | 1 : many | Cascade — deleting user removes all memberships |
| Workspace → WorkspaceMember | 1 : many | Cascade — deleting workspace removes all memberships |
| Workspace → Issue | 1 : many | Cascade — deleting workspace removes all issues |
| User → Issue (assignee) | 0..1 : many | SetNull — unassigns issues when user is removed |
| User → Issue (reporter) | 1 : many | Restrict — reporter must exist |
| User → RefreshToken | 1 : many | Cascade — deleting user revokes all tokens |

## 5. Backend Architecture

### 5.1 Request Lifecycle

Every request flows through the same pipeline:

```
Incoming HTTP
  → JSON body parser + cookie parser + CORS
  → authenticate middleware    (verifies Bearer token → req.user)
  → requireRole middleware     (if route is role-gated → checks WorkspaceMember row)
  → Controller                 (Zod validation → calls service)
  → Service                    (business logic + Prisma queries)
  → Controller                 (formats { data } response)
  → errorHandler               (catches AppError → HTTP status + JSON)
```

### 5.2 Authentication Flow

**Registration:**
1. Validate body with Zod (email, name, password ≥ 8 chars)
2. Check email uniqueness
3. Hash password with bcrypt (cost factor 12)
4. INSERT User
5. Sign `accessToken` (15 min) and `refreshToken` (7 days)
6. INSERT RefreshToken row
7. Return `accessToken` in body, `refreshToken` in HttpOnly cookie

**Token Refresh:**
1. Read `refreshToken` from cookie
2. Verify JWT signature + expiry
3. Look up RefreshToken row in DB (validates it hasn't been revoked)
4. DELETE old RefreshToken row — **rotation, prevents token reuse**
5. Sign new `accessToken` + `refreshToken`
6. INSERT new RefreshToken row
7. Return new `accessToken`; set new cookie

### 5.3 RBAC Middleware

Roles live at the workspace level — a user can be ADMIN in workspace A and MEMBER in workspace B simultaneously.

```typescript
// middleware/requireRole.ts
export const requireRole = (minRole: WorkspaceRole) =>
  async (req: Request, res: Response, next: NextFunction) => {
    const workspace = await prisma.workspace.findUnique({
      where: { slug: req.params.slug }
    });
    if (!workspace) return next(new AppError(404, 'Workspace not found'));

    const member = await prisma.workspaceMember.findUnique({
      where: { workspaceId_userId: { workspaceId: workspace.id, userId: req.user.id } }
    });
    if (!member) return next(new AppError(403, 'Not a member of this workspace'));
    if (minRole === 'ADMIN' && member.role !== 'ADMIN')
      return next(new AppError(403, 'Admin access required'));

    req.workspaceMember = member; // available downstream in controller/service
    next();
  };

// Usage on route
router.post('/', authenticate, requireRole('ADMIN'), createIssue);
```

### 6.2 State Architecture

**Zustand — Auth Store** (persisted to `localStorage`):

```typescript
// store/auth.ts
interface AuthState {
  user: User | null;
  accessToken: string | null;
  login: (token: string, user: User) => void;
  logout: () => void;
}

export const useAuthStore = create<AuthState>()(
  persist(
    (set) => ({
      user: null,
      accessToken: null,
      login: (accessToken, user) => set({ accessToken, user }),
      logout: () => set({ accessToken: null, user: null }),
    }),
    { name: 'auth-store' }
  )
);
```

**React Query — Server State:**

All API data (issues, workspaces, members) is fetched and cached via React Query. Query keys are structured as `['workspace', slug, 'issues']` so targeted invalidation works correctly after mutations. Stale time: 30 s for issue lists, 5 min for workspace metadata.

### 6.3 Axios Interceptor Chain

```typescript
// api/client.ts
const api = axios.create({ baseURL: '/api', withCredentials: true });

// Attach access token to every request
api.interceptors.request.use(config => {
  const token = useAuthStore.getState().accessToken;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// Auto-refresh on 401 + replay original request
api.interceptors.response.use(
  res => res,
  async err => {
    if (err.response?.status === 401 && !err.config._retry) {
      err.config._retry = true;
      try {
        const { data } = await api.post('/auth/refresh');
        useAuthStore.getState().login(data.data.accessToken, data.data.user);
        return api(err.config); // replay original request with new token
      } catch {
        useAuthStore.getState().logout();
        window.location.href = '/auth';
      }
    }
    return Promise.reject(err);
  }
);
```

### 6.4 Kanban Board Implementation

The board uses CSS Grid (one column per `IssueStatus` value). Drag-and-drop via `@dnd-kit/core` calls `PATCH /issues/:id/status` on drop with an optimistic update:

```typescript
const updateStatus = useMutation({
  mutationFn: ({ id, status }: { id: string; status: IssueStatus }) =>
    api.patch(`/workspaces/${slug}/issues/${id}/status`, { status }),

  onMutate: async ({ id, status }) => {
    await queryClient.cancelQueries(['workspace', slug, 'issues']);
    const prev = queryClient.getQueryData(['workspace', slug, 'issues']);
    queryClient.setQueryData(['workspace', slug, 'issues'], (old: Issue[]) =>
      old.map(i => (i.id === id ? { ...i, status } : i))
    );
    return { prev };
  },

  onError: (_, __, ctx) =>
    queryClient.setQueryData(['workspace', slug, 'issues'], ctx?.prev),

  onSettled: () =>
    queryClient.invalidateQueries(['workspace', slug, 'issues']),
});
```

## 7. Module Interaction Map

Cross-module calls always go service → service, never controller → controller.

| Trigger | Caller | Callee | What's Exchanged |
|---|---|---|---|
| User registers | auth.service | prisma (User) | email, name, passwordHash → userId |
| User logs in | auth.service | prisma (User, RefreshToken) | credentials → tokens |
| Create workspace | workspace.service | prisma (Workspace + WorkspaceMember) | Single transaction: create workspace + add creator as ADMIN |
| Create issue | issue.service | workspace.service | slug → workspaceId before INSERT |
| List issues | issue.service | prisma (Issue) | Filters by workspaceId + query params |
| Dashboard summary | dashboard.controller | prisma (groupBy Issue) | Aggregates across user's workspaces |
| Add member | member.service | prisma (User + WorkspaceMember) | findUnique by email, then createMember |
| Token refresh | auth.service | prisma (RefreshToken) | DELETE old, INSERT new — atomic transaction |

### Critical Transaction Boundaries

**Workspace creation** — creating the workspace and adding the creator as ADMIN must be atomic. An orphaned workspace with no admin is unrecoverable.

**Token rotation** — deleting the old refresh token and inserting the new one are wrapped in a transaction. If the insert fails, the old token remains valid and the user is not locked out.

**Issue creation** — counting existing issues (for identifier) and inserting the new issue are wrapped in a transaction to prevent duplicate identifiers under concurrent requests.

### 8.4 Production Checklist

| Item | What to Verify |
|---|---|
| Refresh cookie is `HttpOnly` + `Secure` | Cannot be accessed from JS; HTTPS required |
| JWT secrets are ≥ 32 chars | Short secrets are brute-forceable |
| `DATABASE_URL` uses SSL | Railway Postgres requires `?sslmode=require` |
| CORS origin is locked to production domain | No wildcard `*` in production |
| Prisma uses `migrate deploy` not `migrate dev` | `dev` resets the DB; `deploy` is safe |
| Health check returns 200 within 30 s | Railway restarts service if it fails |
| `GET /api/health` checks DB connectivity | Catches silent DB connection failures |

## 9. System Design Decisions

### Why Single Service, Not Separated Frontend + Backend

Separating into two Railway services adds CORS headers, cross-origin cookie complexity (SameSite=None + Secure), and uses two service slots on a free tier. Serving the Vite build from Express is standard practice (Rails, Django, many Node apps do this). For this assignment it is the correct default unless a CDN is required.

### Why Workspace-Level Roles

A global role per user cannot express "Alice is admin of workspace A and member of workspace B." Workspace-level roles stored in `WorkspaceMember` allow exactly this. The `requireRole` middleware reads from this table on every request — one extra DB query, but always correct without any caching risk.

### Why Refresh Token Rotation

A static refresh token, once stolen, works for its entire lifetime (7 days). With rotation, each use produces a new token and invalidates the old one. If an attacker uses a stolen token, the legitimate user's next refresh fails (their previous token no longer exists in the DB), immediately indicating a compromise. This is the pattern used by Linear, Notion, and GitHub.

### Why Zod for Validation

Zod schemas are the single source of truth for request shape. They generate TypeScript types automatically (`z.infer<typeof schema>`), so controller functions are fully typed without writing separate interface definitions. Validation errors are structured and field-specific, making API debugging straightforward.

### Scalability Notes

The architecture is horizontally scalable without changes. Refresh tokens are in Postgres (not in-memory), so multiple instances share the same token store. Prisma uses a connection pool. Adding Redis for rate limiting or caching would be a straightforward addition without touching any existing module.

### Assignment Requirements Traceability

| Assignment Requirement | Implementation | Location |
|---|---|---|
| Signup / Login | `POST /auth/register`, `POST /auth/login` with bcrypt + JWT | `auth` module |
| Project management | Workspaces CRUD | `workspaces` module |
| Team management | `WorkspaceMember` add / remove / role change | `members` module |
| Task creation + assignment | Issues CRUD with `assigneeId` | `issues` module |
| Status tracking | `PATCH /issues/:id/status` with `IssueStatus` enum | `issues` module |
| Dashboard | `GET /dashboard/summary` + `/my-issues` + `/overdue` | `dashboard` module |
| REST APIs | All endpoints in Section 4 | All modules |
| Database with relationships | Prisma schema with FK constraints + cascades | `schema.prisma` |
| Proper validations | Zod schemas on every POST/PATCH body | All controllers |
| Role-based access control | `WorkspaceRole` enum + `requireRole` middleware | RBAC middleware |
| Railway deployment | `railway.toml` + single-service build pipeline | Section 8 |
